# merco 架构审查报告

> 审查原则：原子能力可靠内聚，可拓展能力清爽干净、边界清晰

---

## 一、架构分层定义

| 层级 | 定义 | 设计要求 |
|------|------|----------|
| **原子能力 (core)** | 缺少则系统无法运行的基础能力 | 可靠、内聚、高效、不可拔插 |
| **可拓展能力 (plugins)** | 可通过插件系统注入的增强能力 | 清爽、边界清晰、可拔插 |

**判断标准**：被核心循环硬依赖 → 原子；有替代方案/可拔插 → 拓展。

---

## 二、核心发现

### 2.1 Agent 是 God Object — 内聚性 2/5

`core/agent.py` 的 `__init__` 直接实例化 15+ 个子系统，跨层导入 sandbox/memory/plugins/hooks/observability/mcp/todo/agents 等。Agent 成了"一切组装车间"而非原子能力本身。

**影响**：原子能力与可拓展能力的边界全部泄漏到 Agent 构造函数中。

### 2.2 Agent Loop 退出条件封闭

`_agent_loop` 硬编码 `if not tool_calls → return content`（agent.py:698），无扩展点。插件无法注入"继续迭代"的意图，Loop Engineering 不可实现。

**本质**：这是 agent loop 的缺陷，不是插件系统的缺陷。原子能力应提供稳定的扩展点。

### 2.3 6 个模块应走插件通道但未走

| 模块 | 当前状态 | 应归属 |
|------|---------|--------|
| `agents/subagent` | Agent 硬初始化 | 可拓展 → SubAgentPlugin |
| `skills/` | Agent 硬初始化 SkillRegistry | 可拓展 → SkillPlugin |
| `mcp/` | Agent 硬初始化 MCPManager | 可拓展 → MCPPlugin |
| `observability/` | Agent 硬初始化 Observer | 可拓展 → ObservabilityPlugin |
| `todo/` | Agent 硬初始化 TodoManager | 可拓展 → 随 SubAgentPlugin |
| `scheduler/` | 未接入，僵尸代码 | 可拓展 → SchedulerPlugin |

### 2.4 Hook 系统是纯通知型，缺少反向控制流

HookRegistry 是 fire-and-forget 事件总线，emit 返回 None，handler 无法影响主流程。架构是单向数据流：

```
Agent ──emit()──→ HookRegistry ──notify──→ Plugin
         单向，插件只能观察，不能控制
```

### 2.5 PluginContext 暴露内部实现细节

| 严重 | 字段 | 问题 |
|------|------|------|
| **致命** | `security_pipeline` | 插件可绕过沙箱 |
| **高** | `recaller` | 直接暴露 HybridRecaller 内部 |
| **高** | `result_pipeline` | 插件可修改所有工具输出 |
| **中** | `add_processor(name, proc)` | 字符串名访问任意属性，等于开放所有 pipeline 写入 |

### 2.6 ToolRegistry 执行流程耦合安全守卫

`registry.execute()` 硬编码了 `tool_guard.check()` 和 `tool_error()`，导致 core 反向依赖 sandbox 和 self_healing 模块。

### 2.7 重复代码

- `memory/compressor.py` 与 `context/processors/compress.py` 功能几乎完全相同
- `sandbox/permissions.py` 与 `guard.py` 功能重叠
- `sandbox/isolation.py` 未被使用

## 三、各模块审查详情

### 3.1 core/ — 原子能力审查

| 模块 | 是否原子 | 内聚性 | 核心问题 | 建议 |
|------|---------|--------|---------|------|

| **config.py** | 部分 | 3/5 | PROVIDER_REGISTRY 硬编码 5 平台；承载可拓展层配置 | 拆 CoreConfig + ExtensionConfig；Provider 注册表外移 |
| **llm.py** | 是 | 4/5 | THINK_TAG_PAIRS 硬编码；surrogate 清理不属于此 | 标签表改为可注册；_clean_surrogates 移到 utils |

### 3.3 plugins/ + hooks/ — 可拓展能力审查

**PluginContext 问题**：
- 暴露 `security_pipeline` → 插件可绕过安全
- `add_processor(name, proc)` 字符串访问任意属性 → 应改为白名单
- 缺少 LLM 请求/响应拦截、Agent Loop 控制、插件间通信等扩展点

**PluginManager 问题**：
- 缺少 `unregister()`、依赖声明/解析、冲突检测、健康检查
- `activate_all()` 时序问题：TodoManager/SubAgentManager 在 activate 后才赋值，已激活插件拿不到

**HookRegistry 问题**：
- 纯 fire-and-forget，无返回值、无异常隔离、无优先级、无短路机制
- 预定义 hook 文件是空壳占位，只增加调用开销
- 缺少关键 hook 入口：`llm.before_chat`、`llm.after_chat`、`conversation.turn`、`plugin.*`

### 3.4 其他模块归属

| 模块 | 归属 | 关键问题 |

| `skills/` | 可拓展 | Agent 不加载 skill 也能运行 |
| `mcp/` | 可拓展 | 可选依赖，天然插件场景 |
| `gateway/` | 可拓展 | TODO 空壳，每个网关一个插件 |

## 四、修复方案

### 4.1 Agent Loop 开放扩展点 — LoopPolicy

**问题**：退出条件封闭，插件无法注入"继续迭代"意图。

**方案**：引入 `LoopPolicy` 协议，默认行为复刻当前逻辑，插件可注册自定义策略。

```python
class LoopPolicy(ABC):
    def should_continue(self, response: dict, state: LoopState) -> bool: ...
    def should_exit(self, response: dict, state: LoopState) -> bool: ...

class DefaultLoopPolicy(LoopPolicy):
    """当前行为：无 tool_call → 退出"""
    def should_continue(self, response, state):
        return bool(state.pending_tool_calls)
    def should_exit(self, response, state):
        return not state.pending_tool_calls and state.iteration > 0
```

**Tradeoff**：
- 改动小（`_agent_loop` 末尾 +5 行，新增 LoopPolicy 协议 ~20 行）
- 零破坏（默认策略复刻当前行为）
- 迭代上限复用 `_max_tool_calls`，无无限循环风险
- 可从 LoopPolicy 自然演进到 Command 对象模式

### 4.2 拆分 Agent God Object — 依赖注入

**问题**：Agent.__init__ 组装 15+ 子系统，边界全部泄漏。

**方案**：引入 `AgentFactory`（或利用 `merco/setup.py`），Agent 只接收已组装好的依赖。

```python
# 之前
class Agent:
    def __init__(self):
        self.mcp_manager = MCPManager(...)    # 硬初始化
        self.todo_manager = TodoManager(...)   # 硬初始化
        self.observer = Observer(...)          # 硬初始化

# 之后
class Agent:
    def __init__(self, llm, session, hooks, guard, **deps):
        self.llm = llm
        self.session = session
        self.hooks = hooks
        self.guard = guard
        # 可拓展能力通过 deps 注入，Agent 不感知具体类型
```

**Tradeoff**：
- 改动中等，需重构所有 `Agent()` 构造点
- 对插件开发者体验提升明显：不再需要理解 Agent 内部结构
- 可分阶段：先拆可拓展模块（6 个插件化模块），再拆组装逻辑

### 4.3 HookRegistry 支持反向控制流 — HookResult

**问题**：fire-and-forget 无法让插件影响主流程。

**方案**：emit 返回 `HookResult`，handler 可返回控制信号。

```python
class HookResult:
    continue_: bool = True     # 继续后续 handler
    short_circuit: Any = None  # 短路主流程
    inject_message: str = None # 向 agent loop 注入消息

async def emit(self, event: str, **kwargs) -> HookResult:
    result = HookResult()
    for handler in self._hooks.get(event, []):
        r = await handler(**kwargs) if ... else handler(**kwargs)
        if isinstance(r, HookResult):
            result.merge(r)
            if not result.continue_:
                break
    return result
```

**Tradeoff**：
- 改动中等（emit 签名变更，~20 调用点）
- 向后兼容（现有 handler 不返回值 → None → 无副作用）
- 核心风险：副作用扩散、多 handler 竞态 → 需优先级+白名单机制

### 4.4 ToolRegistry 安全守卫解耦 — 中间件链

**问题**：registry.execute() 硬编码 guard + error handling → core 反向依赖 sandbox。

**方案**：将 execute 拆为中间件链，registry 只做路由。

```python
# registry 只负责路由
async def execute(self, tool_name, **kwargs):
    tool = self.get(tool_name)
    if not tool:
        return {"error": f"工具 '{tool_name}' 不存在"}
    return await self._middleware_chain(tool, **kwargs)

# 中间件由外部注册
registry.use(GuardMiddleware(guard))
registry.use(ErrorHandlingMiddleware())
```

**Tradeoff**：
- 改动中等（registry.execute 重构 + 中间件定义 ~50 行）
- 解除 core → sandbox 反向依赖
- edit.py 的 confirm_edit / snapshot 同理移到中间件

### 4.5 PluginContext 安全加固

**问题**：暴露 security_pipeline，add_processor 字符串访问任意属性。

**方案**：
- 移除 `security_pipeline` 直接暴露，改为只读接口（`get_security_rules()` 而非 pipeline 本身）
- `add_processor` 改为白名单模式：

```python
_PIPELINE_WHITELIST = {"result_pipeline", "recovery_pipeline", "memory_save_pipeline", "context_pipeline"}

def add_processor(self, pipeline_name: str, processor) -> None:
    if pipeline_name not in _PIPELINE_WHITELIST:
        raise ValueError(f"Pipeline '{pipeline_name}' not extensible")
    pipeline = getattr(self, pipeline_name, None)
    if pipeline and hasattr(pipeline, 'use'):
        pipeline.use(processor)
```

**Tradeoff**：改动极小，安全性大幅提升。

## 六、目标架构

```
┌─────────────────────────────────────────────────────┐
│                  原子能力 (core)                      │
│                                                      │
│  Agent (核心循环)    Session    LLM Client            │
│  LoopPolicy (扩展点) Context   Message               │
│  ToolRegistry (路由+中间件链)  BaseTool              │
│  HookRegistry (HookResult)    Pipeline 框架          │
│  Plugin 基类+PluginManager   SessionStore            │
│  MemoryBackend+Store+SavePipeline                    │
│  Sandbox Guard+Snapshot    Config (CoreConfig)       │
│                                                      │
├─────────────────────────────────────────────────────┤
│                  可拓展能力 (plugins)                  │
│                                                      │
│  ObservabilityPlugin   SkillPlugin   MCPPlugin       │
│  SubAgentPlugin(+Todo) WebPlugin     SchedulerPlugin │
│  LoopEngineeringPlugin (LoopPolicy 注入)             │
│  GatewayPlugin (Discord/Telegram)                    │
│  自定义 MemoryStrategy / ContextProcessor            │
│                                                      │
├─────────────────────────────────────────────────────┤
│                  内置工具 (原子)                       │
│  BashTool  ReadFile  WriteFile  EditFile             │
│                                                      │
├─────────────────────────────────────────────────────┤
│                  内置插件 (builtin)                    │
│  SuperpowerPlugin  (示例)                             │
└─────────────────────────────────────────────────────┘
```

**设计原则**：
- 原子能力之间通过接口依赖，不通过实现依赖
- 可拓展能力通过 PluginContext 的白名单接口注入，不暴露内部结构
- HookRegistry 同时支持通知型和拦截型事件，提供反向控制流通道
- Agent Loop 通过 LoopPolicy 开放退出策略，插件可注册自定义迭代规则
